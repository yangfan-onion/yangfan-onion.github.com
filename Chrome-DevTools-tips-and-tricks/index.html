Chrome DevTools tips and tricks

Apr 07, 2016

Elements

  1. 选择元素按H可以显示和隐藏当前元素，省去了手动添加(display:none;)操作
  2. 为元素添加Subtree modification, Attribute modification, Node removal监听
  3. Event Listeners 可以查看当前选中元素有事件，可以绑定到祖先元素上的事件，并且可以查看事件的代码
  4. Toggle Element state可以查看当前元素的:active, :focus, :hover, :visited样式
  5. Ctrl+F支持子节点查找 form input

Console

  • $0 => 选择当前选中的元素， 右键可以修改
  • $(‘img’) => document.querySelector
  • $$(‘img’) ＝> document.querySelectorAll
  • console.table([{a:1, b:2, c:3}, {a:”foo”, b:false, c:undefined}]);
  • dir($0)

  • debug(functionName) && undebug 控制台中打断点

  • profile(“My profile”) && profileEnd(“My profile”) 用于比较耗费CPU的操作

Sources

  1. 可以在代码中添加debugger; 程序在执行时会在此断点
  2. 动态加载的JS代码调试， 代码头部加入 // @sourceURL=example.js 通知DevTools
  3. Ctrl + P可以查找任意文件
  4. Ctrl + P + : + 行号跳转到文件相应行
  5. Ctrl+Shift+O可以跳转到任意方法
  6. 左下角的{}可以格式化压缩的JS,CSS方便查看代码调试
  7. 添加断点调试JS, 添加条件断点
  8. 单步跳过，进入，跳出方法, black box, 跳过jquery, angular的源代码，只在我们自己写的代码中调试
  9. 添加表达式
  10. 函数的调用栈

Network

  1. Filter可以多选
  2. size/content可以进行优化判断 size:文件传输大小, content:实际大小; size=Gzip(文件大小)+response header的大小
  3. 非二进制文件的size等于content说明未开启Gzip, 需要优化
  4. size大于content, 一般是小图片，可能20B大小，size中还包含了response_headers的大小，所以切图的时候，可以把多个小图片做成一张雪碧图，一是减少网络请求，二是可以减少过多response header的传输浪费
  5. time/latency => 总时间/等待时间 开始时间到TTFB(首字节加载时间)就是等待时间
  6. initator 资源是由谁请求的， 解释器或者自定义的JS，由JS发起的会存在一些延迟
  7. HAR(HTTP Archive)前后台开发人员协作优化性能问题 http://ericduran.github.io/chromeHAR/
  8. DOMContent event Fire（解释完当前页面）蓝线， Load Event Fire（加载完所有资源）红线

如何优化Queueing， Stalled
  1. 资源优先级 图片的优先级一般是低于html, css, js的，可以在Network的选项卡里调出优先级一列
  2. 同一域名网络请求并发数 chrome 6个

排队时间高就是因为并发的6个链接正在处理优先级高的资源，要等到优先级高的资源处理完了，在复用这些链接请求优先级更低的资源
这个时候我们将图片换到另外的域名下，在同一优先级的图片下面，logo.png是会优先排队的，这样就可以避免排队时间的过多的消耗，

鱼与熊掌不可兼得，放到另外的域名下，DNS查询也是会增加消耗的，一般2～4个为宜

如何优化TTFB

TTFB 是一个可以优化的点，利用CDN, 让内容距离用户更近，更快的服务器响应

如何优化Content Download

减小内容大小；
特别是在首次加载，TCP连接在开始是很慢的，window size是很小的，~14kb, 超过14kb的内容需要另外的TCP包进行传输，会增加一个round trip time(来回时间)，在来回时间很长的情况下，会拖慢首次加载的速度
